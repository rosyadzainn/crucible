Critical Line 05

CASCADE drop on primary key

DROP INDEX users_pkey CASCADE silently removes every foreign key and view depending on it. No backup taken first. One typo here is unrecoverable.

Critical Line 04

Snapshot stored on volatile path

/tmp is wiped on every reboot. If recovery requires a restart — which it often does — this snapshot is gone. Use a persistent, off-host destination.

High Line 01

Root SSH bypasses audit trail

ssh root@… leaves no who/what/when record. Use sudo with session logging so every action stays attributable to a human.

High Line 03

Primary stopped before replica promotion

Application sees a hard outage window between the stop and the promotion. Sequence should be: promote replica → drain connections → stop primary.

High Cross-line · between 04 & 05

No backup verification before destruction

The runbook drops an index on production data without verifying that the snapshot from step 04 is restorable. An untested backup is a hope, not a backup.

Medium Line 06

Port-listen check ≠ service health

nc -z localhost 5432 confirms the socket is open, not that Postgres accepts queries. A real SELECT 1 probe catches the difference.

Medium Cross-line · steps 02–07

No rollback plan

Every step assumes forward progress. There's no documented "if this fails, do this" — escalation reverts to tribal knowledge under pressure.
